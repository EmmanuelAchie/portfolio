The Problem

Why does this exist?

Sports analytics is increasingly data-driven, but raw player statistics are often stored in flat files or spreadsheets — formats that make cross-season comparisons, multi-club queries, and performance trend analysis difficult. A flat CSV with 125 player rows quickly becomes unmanageable when you want to answer questions like: which players maintained consistent goal contributions across multiple clubs? or which teams had the highest average goals per player in a given season?

The goal of this project was to design a properly normalized relational database that could answer these questions efficiently through structured queries, and that could scale gracefully as new seasons, leagues, or statistical categories are added over time.

The core design challenge: build a schema flexible enough to accommodate future expansion without requiring structural rewrites, while keeping join complexity low enough that queries remain readable and performant.

-- Example: Which club has the highest average goals per player? SELECT c.club_name, AVG (ps.goals) AS avg_goals_per_player FROM player_stats ps JOIN players p ON ps.player_id = p.player_id JOIN clubs c ON p.club_id = c.club_id GROUP BY c.clubs, c.club_name HAVING AVG (ps.goals) = (SELECT MAX(avg_goals) FROM (SELECT AVG(ps2.goals) AS avg_goals FROM player_stats ps2 JOIN players ps2 ON ps2.player_id = p2.player_id GROUP BY p2.club_id) AS sub);
🔑
Primary Keys
Each table has a surrogate primary key, ensuring rows remain independently addressable even when natural identifiers (like player names) change or repeat.
🔗
Foreign Keys
Stats connect to Seasons, which bridges Players and Clubs — a deliberate join chain that allows a single player to have records at multiple clubs across different years.
📈
Scalability
New stat categories (e.g. xG, progressive carries) can be added as columns to the Stats table without touching Players, Clubs, or Seasons. No structural rewrites needed.
03
My Process

How I built it

1

Define the Entities

I started by identifying the core real-world objects: players, clubs, player stats, and player bio. Rather than storing everything in one flat table, I mapped out what belonged together and what needed to be separated to avoid data redundancy.

2

Normalize to 3NF

I applied Third Normal Form to eliminate transitive dependencies, for example, league and country are properties of a club, not a player, so they live in the Clubs table. This prevents update anomalies if a club's league changes.

3

Populate & Query Test

I populated all 4 tables with 125 rows of realistic player data, then wrote and tested a range of queries — from simple lookups to multi-join aggregations — to verify the schema answered real analytical questions efficiently.

04
Takeaways

What I learned

  • Good schema design is about anticipating the questions you will want to ask later,not just storing data you have today. Building with future queries in mind changed how I structured every table.
  • Normalization trades write simplicity for read power. A flat table is easy to fill, but a normalized schema lets you query from any angle without duplication or inconsistency.
  • Join logic is easier to reason about when each table has a single, clear responsibility.When I was confused about where a field belonged, that confusion was a signal the design needed more thought.
