The Challenge

47 pages. One system.

Building a single-page website is straightforward. Building 47 interconnected pages that all feel like they belong to the same product without a framework or component library is a real engineering and design challenge.

The project brief was to create an interesting interactive website and my site allows users to navigate from a homepage down through continent pages and into individual country profiles. The core technical challenge: how do you maintain visual consistency, predictable navigation, and a cohesive aesthetic across a site of this scale using only vanilla HTML, CSS, and JavaScript?

This required thinking in systems from the start, not just building individual pages, but designing a reusable structure that could be applied consistently at every level of the hierarchy, from the global homepage to the deepest country detail page.

§ 02b — Technical Highlights

Key Interactions & Features

🃏
Card-Flip Animations (CSS)
Country cards on continent pages use CSS 3D transforms to flip on hover, revealing key facts on the reverse side — adding interactivity without JavaScript overhead.
🖼️
JavaScript Image Slideshows
Each country detail page includes a JS-powered image slideshow cycling through local attractions — built from scratch with vanilla JS event listeners and DOM manipulation.
🗺️
Consistent Design System
A shared CSS file defines typography, spacing, color palette, and component styles. Every page inherits this system — ensuring visual coherence across all 47 pages without repetition.
📱
Responsive Layouts
Media queries adapt every tier of the hierarchy — homepage, continent grids, and country profiles — for mobile and tablet viewports without layout breakage.

My Process

How I built it

1

Information Architecture First

Before writing a single line of HTML, I mapped the full site hierarchy — homepage, 5 continent pages, 40+ country profiles — and planned the folder structure and file naming conventions.

2

Basic HTML with Template-Driven Development

I wrote the basic HTML stucture for the homepage and individual continent page as well as the content to go with those pages. I then wrote the HTML for the country pages. Rather than building each of the 40+ country pages from scratch, I built one country page template and replicated it with different content.

3

Design System with CSS

I built the shared CSS stylesheet, defining color variables, typography scale, spacing units, and reusable component classes before designing any individual page. This meant every page I built after had a consistent visual foundation from line one.

4

Interactive Feature Layer

With the static structure solid, I layered in JavaScript features the image slideshow and card-flip animations. Separating structure (HTML) from style (CSS) from behavior (JS) made debugging and iteration much cleaner throughout the build.

04
Challenges

What was hard

  • Scale
    Maintaining visual consistency across 47 pages without a framework was the core challenge. The solution was investing heavily in the shared CSS system upfront — one source of truth for all visual decisions meant changes propagated everywhere automatically.
  • Navigation
    Relative file paths across a three-tier directory structure were genuinely tricky. A broken link on a country page might be two or three levels deep. Establishing consistent folder conventions from the start was essential to keeping paths predictable.
  • Interactivity
    Building a JavaScript slideshow from scratch, without a library like Swiper or Slick, required understanding DOM manipulation, event listeners, and timing logic directly. It was harder than using a library, but the understanding gained was much deeper.
  • Responsive
    Making 47 pages responsive meant the template had to be responsive. If the base template had layout bugs on mobile, those bugs multiplied across every page. Testing the template thoroughly before scaling saved significant rework time.
05
Takeaways

What I learned

  • Design systems are not just for large teams, even solo projects benefit enormously from a single source of visual truth. Every minute spent defining CSS variables upfront saved five minutes of hunting down inconsistencies later.
  • Information architecture is a front-end skill, not just a UX skill. The way you structure your files, folders, and page hierarchy directly determines how maintainable and navigable your codebase is months later.
  • Building JavaScript features from scratch, without libraries, is the fastest way to genuinely understand how the web works. I understood DOM manipulation and event-driven logic more deeply after this project than I had from any tutorial.
  • At scale, templates are everything.The real leverage in this project was not building 47 pages, it was building one great template and applying it 47 times. That principle applies directly to component-based frameworks like React.
